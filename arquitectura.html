Módulo 4: Arquitectura por Capas y Validación

📌 Introducción

Un proyecto profesional separa la lógica de enrutamiento (Controller), la manipulación de datos (Model) y la presentación (View). Además, la validación de datos debe ser centralizada, reutilizable y ocurrir estrictamente en el servidor, independientemente de cualquier validación en JavaScript.

4.1 Flujo de Datos MVC en la Práctica

  1. Request: El usuario envía un POST a /usuarios/crear.
  2. Controller: UsuarioController@crear recibe el $_POST. Llama a la clase Validator. Si falla, redirige con "Flash Messages". Si pasa, llama al Model.
  3. Model: UsuarioModel recibe los datos limpios, hashea la contraseña y ejecuta la consulta PDO preparada.
  4. View: El Controller carga views/usuarios/exito.php, pasando solo las variables necesarias para renderizar.

4.2 Clase Validadora Reutilizable

En lugar de repetir if (empty(...)), creamos una clase estática que acumula errores y valida reglas comunes.

namespace App\Utils;

class Validator {
    private static array $errors = [];

    public static function isRequired(string $field, mixed $value): bool {
        if (empty(trim((string)$value))) {
            self::$errors[$field] = "El campo $field es obligatorio.";
            return false;
        }
        return true;
    }

    public static function isEmail(string $field, string $value): bool {
        if (!filter_var($value, FILTER_VALIDATE_EMAIL)) {
            self::$errors[$field] = "El campo $field debe ser un correo electrónico válido.";
            return false;
        }
        return true;
    }

    public static function isCedula(string $field, string $value): bool {
        // Ejemplo: Validación de cédula venezolana (V-12345678 o E-12345678)
        if (!preg_match('/^[VEve]-?\d{7,8}$/', $value)) {
            self::$errors[$field] = "Formato de cédula inválido (Ej: V-12345678).";
            return false;
        }
        return true;
    }

    public static function getErrors(): array {
        return self::$errors;
    }

    public static function isValid(): bool {
        return empty(self::$errors);
    }
    
    public static function clear(): void {
        self::$errors = [];
    }
}

Uso en el Controlador:

Validator::clear();
Validator::isRequired('email', $_POST['email']);
Validator::isEmail('email', $_POST['email']);
Validator::isCedula('cedula', $_POST['cedula']);

if (!Validator::isValid()) {
    // Guardar errores en sesión (Flash Message) y redirigir
    $_SESSION['errors'] = Validator::getErrors();
    $_SESSION['old_input'] = $_POST; // Para repoblar el formulario
    header("Location: /registro");
    exit;
}

4.3 Middleware de Control de Acceso por Roles (RBAC)

Los middlewares interceptan la petición antes de que llegue a la lógica del controlador.

namespace App\Middleware;

class RoleMiddleware {
    public static function handle(array $allowedRoles): void {
        if (!isset($_SESSION['user_id'])) {
            header("Location: /login");
            exit;
        }

        if (!in_array($_SESSION['role'], $allowedRoles, true)) {
            http_response_code(403);
            // Cargar vista de "Acceso Denegado"
            require __DIR__ . '/../../app/Views/errors/403.php';
            exit;
        }
    }
}

// Uso en el Controller:
// RoleMiddleware::handle(['admin', 'supervisor']);

4.4 Depuración Profesional (Debugging) y Logs

Usar var_dump() y die() es aceptable en desarrollo local, pero inaceptable en producción. Debemos configurar un sistema de logs.

// config.php (Entorno de Producción)
ini_set('display_errors', 0);          // NUNCA mostrar errores al usuario final
ini_set('log_errors', 1);              // Activar registro en archivo
ini_set('error_log', __DIR__ . '/../storage/logs/app_errors.log');

// Función helper para logging estructurado
function log_audit(string $action, array $context = []): void {
    $logEntry = sprintf(
        "[%s] %s | User: %s | IP: %s | Details: %s\n",
        date('Y-m-d H:i:s'),
        $action,
        $_SESSION['user_id'] ?? 'GUEST',
        $_SERVER['REMOTE_ADDR'],
        json_encode($context)
    );
    error_log($logEntry, 3, __DIR__ . '/../storage/logs/audit.log');
}

// Uso:
// log_audit('INTENTO_LOGIN_FALLIDO', ['email' => $_POST['email']]);

💡 Conclusiones y Consejos Profesionales

  • Flash Messages: Son mensajes de sesión que se borran inmediatamente después de leerse una vez. Ideales para "Registro exitoso" o "Error de validación".
  • Subida de Archivos: Nunca confíes en $_FILES['archivo']['type'] (el cliente puede falsificarlo). Usa finfo_file() para verificar el MIME type real del contenido del archivo. Guarda los archivos fuera de /public o con nombres aleatorios (UUID) sin extensión ejecutable.
  • Principio de Menor Privilegio: La cuenta de usuario de la base de datos que usa PHP solo debe tener permisos SELECT, INSERT, UPDATE, DELETE en las tablas necesarias. Nunca usar el usuario root en producción.
