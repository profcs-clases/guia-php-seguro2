Módulo 2: Conexión a Base de Datos Avanzada

📌 Introducción

La interacción con la base de datos es el vector de ataque más común (SQL Injection) y el cuello de botella de rendimiento más frecuente. Este módulo profundiza en por qué PDO es el estándar de la industria, cómo implementar el patrón Singleton correctamente y la importancia crítica de desactivar la emulación de consultas preparadas.

2.1 PDO vs MySQLi: Análisis Técnico

Ambas extensiones son seguras si se usan con consultas preparadas, pero PDO ofrece ventajas arquitectónicas decisivas para proyectos escalables.

Característica PDO (Recomendado) MySQLi
Soporte de Motores 12+ (MySQL, PostgreSQL, SQLite, SQL Server) Solo MySQL / MariaDB
Parámetros Nombrados ✅ Sí (:email). Mejora legibilidad en queries complejas. ❌ No (Solo ? posicional)
Orientación a Objetos ✅ 100% Nativo y consistente. ⚠️ Híbrido (mezcla estilos procedural y OO, propenso a errores).

2.2 Implementación Robusta del Patrón Singleton

El Singleton garantiza una única instancia de la conexión en toda la ejecución del script, ahorrando recursos. La clave de la seguridad está en las opciones de PDO.

namespace App\Models;
use PDO;
use PDOException;

class Database {
    private static ?self $instance = null;
    private PDO $connection;

    // Constructor privado para evitar instanciación externa (new Database())
    private function __construct() {
        $dsn = "mysql:host=" . DB_HOST . ";dbname=" . DB_NAME . ";charset=utf8mb4";
        
        $options = [
            PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION, // Lanzar excepciones en errores
            PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,       // Devolver arrays asociativos por defecto
            PDO::ATTR_EMULATE_PREPARES   => false,                  // ⚠️ CRÍTICO: Fuerza al motor de BD a preparar la query, no a PHP.
        ];

        try {
            $this->connection = new PDO($dsn, DB_USER, DB_PASS, $options);
        } catch (PDOException $e) {
            // En producción, registrar en log y mostrar mensaje genérico
            error_log("Fallo crítico de conexión a BD: " . $e->getMessage());
            throw new \Exception("Error de conexión al sistema. Intente más tarde.");
        }
    }

    public static function getInstance(): self {
        if (self::$instance === null) {
            self::$instance = new self();
        }
        return self::$instance;
    }

    public function getConnection(): PDO {
        return $this->connection;
    }
}

🕵️‍♂️ Caso de Hackeo: SQL Injection por Emulación

Si PDO::ATTR_EMULATE_PREPARES está en true (valor por defecto en algunos entornos), PHP simula la preparación. En ciertos edge cases con conjuntos de caracteres extraños, un atacante puede manipular la codificación para "escapar" de las comillas e inyectar SQL, incluso usando prepare().

🛡️ La Defensa: Desactivar Emulación y Usar Named Parameters

Al establecer ATTR_EMULATE_PREPARES => false, la consulta se envía al motor de base de datos en dos pasos distintos: primero la estructura (compilada), luego los datos (como valores literales, no como código). Es matemáticamente imposible que los datos se ejecuten como SQL.

$db = Database::getInstance()->getConnection();
$stmt = $db->prepare("SELECT id, nombre FROM usuarios WHERE email = :email AND estado = :estado");
// execute recibe un array asociativo que coincide con los named parameters
$stmt->execute(['email' => $_POST['email'], 'estado' => 'activo']);
$usuario = $stmt->fetch();

2.3 Transacciones ACID para Integridad de Datos

Cuando una operación lógica requiere modificar múltiples tablas (ej: crear un usuario y su registro de auditoría), deben ejecutarse dentro de una transacción. Si una falla, se revierten todos los cambios (ROLLBACK).

try {
    $db->beginTransaction();

    $stmt1 = $db->prepare("INSERT INTO usuarios (email, password_hash) VALUES (?, ?)");
    $stmt1->execute([$email, $hash]);
    $userId = $db->lastInsertId();

    $stmt2 = $db->prepare("INSERT INTO auditoria (user_id, accion) VALUES (?, ?)");
    $stmt2->execute([$userId, 'REGISTRO_CREADO']);

    $db->commit(); // Confirma los cambios permanentemente
} catch (Exception $e) {
    $db->rollBack(); // Deshace TODO si cualquier query falla
    error_log("Error en transacción de registro: " . $e->getMessage());
    throw $e;
}

💡 Conclusiones y Consejos Profesionales

  • Nunca uses interpolación de strings ("SELECT * FROM users WHERE id = $id") bajo ninguna circunstancia.
  • Usa utf8mb4 como charset, no utf8. utf8mb4 soporta emojis y caracteres Unicode completos, previniendo vulnerabilidades de codificación.
  • El método lastInsertId() debe llamarse inmediatamente después del INSERT y dentro de la misma conexión, ya que es específico de la sesión.
